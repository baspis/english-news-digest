# 英語ニュース学習アプリ再設計 — 引き継ぎブリーフ

## 1. 依頼の目的

既存のプロトタイプ `english-news-digest` を、**大上段の設計から作り直したい**。

現状は「動くもの」はあるが、情報設計・画面構成・日次運用の前提が固まっていない。  
**設計フェーズ**として、以下を出してほしい:

1. プロダクトの目的とスコープ定義
2. 画面構成（3画面ナビゲーション）と各画面の責務
3. データモデル（日付・記事・学習コンテンツの関係）
4. ビルドパイプライン（取得 → 選定 → 解析 → 出力）の再設計
5. ファイル/URL 構造
6. MVP と将来拡張の切り分け
7. 未決事項と推奨判断（理由付き）

**このフェーズではコード実装は不要。** 設計書・画面ワイヤ・データスキーマ・運用フローを主成果物とする。

---

## 2. ユーザー背景

- 41歳、日本在住。英語学習は **fluency-first**（IELTS 8.0 は計測目標であり目的ではない）
- 現在地（2026-05 EF SET）: Reading/Listening **B2**、Speaking/Writing **B1**
- 学習の柱: Input / Output / Language Focus / Fluency の4本柱を維持
- 省力化重視。iPhone で読む想定。追跡やツール構築に時間をかけたくない
- 健康・睡眠・家族・仕事を優先。英語学習は毎日少し、構造は崩さない

---

## 3. プロダクトの本質（変えないこと）

**「英語ニュースを集める」だけではない。**  
**B2 学習者が、毎日少しずつニュース英文を読み、語彙・文法・訳を得て、必要なら Anki に流せる**ことが目的。

- ソースは無料の英語ニュース（現状: Japan Today, BBC, Guardian の RSS）
- **日本関連ニュースを主軸** + 国際重要ニュースを補助
- NHK は品質面で採用見送り（BBC / Japan Today 中心）
- 1日あたりの記事数は **固定5件** を前提にしたい（多すぎると学習にならない）

---

## 4. 望む UX（これが今回の核心）

**3段階ナビゲーションに統一する:**

```
[1] カレンダー
      ↓ 日付を選択
[2] その日の記事一覧（常に5件）
      ↓ 記事を選択
[3] 記事詳細（学習画面）
```

### 画面1: カレンダー（トップ）

- 月表示カレンダー
- **記事が生成された日だけ**視覚的に区別（クリック可能）
- 今日の日付をハイライト
- 前月・翌月への移動（将来でも可だが、設計には含める）
- ここには記事本文や解説は出さない。**日付選択の入口だけ**

### 画面2: 日次記事一覧

- 選択した日付の **記事をちょうど5件** 表示
- 各カードに最低限: タイトル、ソース、カテゴリ（Japan / World）、日本語要約（1〜2文）、元記事リンク
- カードタップで詳細画面へ
- パンくず: `カレンダー ← 2026-06-09`
- **5件未満・5件超過は設計上どう扱うかを決めること**（後述「未決事項」参照）

### 画面3: 記事詳細（学習画面）

- 英文を文単位で表示
- 各文に: 日本語訳、文法メモ（B2 向け）
- 記事全体: 語彙リスト（IPA、意味、例文、例文訳）、文法ポイント
- 元記事へのリンク
- パンくず: `カレンダー ← 2026-06-09 ← 記事タイトル`
- **「通常版」と「深掘り版（文節分解）」を同一画面内でどう扱うか**は設計課題（別ページ / タブ / 折りたたみ など）

## 5. 現状プロトタイプの問題（設計から直したい点）

| 問題 | 詳細 |
|------|------|
| 記事数が不安定 | 「今日 JST で公開されたものを全部」になり、5件にならない日もある。途中ビルドで5件/9件/16件が混在 |
| 画面の責務が曖昧 | カレンダー・日次・詳細はあるが、**「1日=5件」が設計の軸になっていない** |
| 通常版と深掘り版が分裂 | `--deep` で別 HTML・別キャッシュ。日次ページから「詳細版」リンクが生えるなど、UX が二重 |
| 本文ノイズ | Japan Today 取得時に広告・フッター・SNS 案内が本文に混入し、AI がそれも「学習文」として解析 |
| ビルドが遅い・不透明 | 1記事あたり cursor-agent で1〜3分。16件直列は非現実的。進捗も見えにくい |
| データと表示の不整合 | `data/` に9件キャッシュ、`dist/` に5件 HTML など、途中状態が残りやすい |
| 実装が1ファイル肥大 | `build_learning.py` が取得・AI・HTML生成・Anki・カレンダーを全部担当 |

**「画面を足す」ではなく、日次エディション（date → exactly 5 articles → detail）を中心に据え直してほしい。**

## 6. 既存資産（再利用候補）

設計時に「捨てる/残す」を明示すること。

### 技術スタック（現状）

- Python 3（依存パッケージほぼなし）
- RSS 取得: `urllib` + `xml.etree`
- 本文取得: **Obscura**（`~/.local/bin/obscura`）— JS レンダリングページ向け
- AI 解析: **cursor-agent** CLI（`~/.local/bin/cursor-agent`、デフォルト `composer-2.5`）
- 出力: 静的 HTML + JSON キャッシュ
- Anki 連携: `english-anki` プロジェクト（TSV 正本 → `import.py` → AnkiConnect）

### 学習コンテンツのスキーマ（現行）

AI 出力 JSON の概形:

```json
{
  "summary_ja": "...",
  "sentences": [
    {
      "text": "English sentence",
      "translation_ja": "...",
      "grammar_ja": "...",
      "chunks": [{"text","role_ja","literal_ja","note_ja"}],
      "deep_dive_ja": "..."
    }
  ],
  "vocabulary": [{"word","pronunciation","meaning_ja","example","example_ja"}],
  "grammar_points": [{"id","title","rule","example","meaning_ja"}]
}
```

Anki TSV は `english-anki/materials/news-digest/batches/{date}-vocabulary.tsv` 等。

## 7. 設計で決めてほしい未決事項

### A. 「1日5件」の定義

候補（推奨を1つ選び、理由を書くこと）:

1. **厳密5件**: 選定アルゴリズムで常に5件。不足時は前日繰越や別ソース補充はしない（空欄 or ビルド失敗）
2. **最大5件**: 今日公開が3件なら3件だけ表示
3. **常に5件**: 不足時は直近の未収録記事で埋める

ユーザー意向は **「一覧は5件」** だが、ソースの都合で毎日5本揃うとは限らない。**この矛盾の解き方が設計の要**。

### B. 5件の内訳

- 例: Japan 3 + World 2、Japan 5 + World 0、など
- 選定基準: 公開日時（JST）、ソース優先度、重複排除、カテゴリバランス

### C. 深掘り解説の位置づけ

- 5件すべて深掘りか、1〜2件だけか
- コスト: 深掘り1件あたり AI 2〜4分、JSON 約3倍
- 同一記事の「通常」「深掘り」をユーザーが意識しない UX にできるか

### D. ビルドのタイミングと場所

- 手動（Mac）vs VPS cron（`llm-devbox`）vs 両方
- 1日1回のバッチか、記事追加の増分か
- 失敗時のリトライ・部分成功の扱い

### E. 公開方法

- ローカル `file://` で開く（現状）
- Cloudflare Pages 等の静的ホスティング
- iPhone から読む導線

### F. Anki 連携

- 毎日自動 import か手動か
- 5件分の語彙・文法を1バッチにまとめる現行方式でよいか
- 重複排除ルール（同じ単語が複日に出た場合）

## 8. 制約・非機能要件

| 項目 | 要件 |
|------|------|
| 無料ソースのみ | 有料ペイウォール記事は対象外 |
| 英語レベル | B2 読解向け。訳・文法メモは日本語 |
| 速度 | 初回ビルドは遅くてよいが、**2回目以降はキャッシュで速く** |
| 可観測性 | どの段階で何件成功/失敗したかログで分かること |
| シンプルさ | Year 1 英語学習計画に「新しいコアツール」を増やさない |
| プライバシー | 記事本文を外部 AI に送る（cursor-agent）。設計上明示すること |

## 10. 成功基準（設計が良いかの判定）

- ユーザーが迷わず **カレンダー → 5件一覧 → 詳細** に辿れる
- 「今日何を読むか」が **日付を開けば即わかる**（5件固定 or 不足時のルールが明確）
- ビルドパイプラインが **モジュール分割** され、1ファイル890行に戻らない
- 深掘り解説が **プロダクトの主価値** として位置づけられ、コストと両立する
- 既存の `english-anki` 連携を壊さない（または移行手順がある）
- 実装者（別セッションの Agent）が、この設計だけでコーディングを開始できる

## 11. 意図的にスコープ外（今回の設計に入れない）

- ユーザーアカウント / ログイン
- 読了状態の同期（複数デバイス）
- Speaking/Writing の練習 UI（ニュースは Input + Language Focus 向け）
- 新しい英語学習ツールの追加（ELSA 等との統合は別プロジェクト）
